java基础

java语言跟其他语言对比起来，比如python、php来讲，其实是相当的啰嗦。

对比php，php本身是可以面像过程编程的，有函数、全局作用域。传递变量，可以使用全局变量来传递。最近看了一些java的apache commons模块，发现其提供的一些功能，php本身的系统库函数就自带。在java中还要引入库使用。（这是大缺点吗？就是引入一个包而已，貌似也不是什么缺点）。

java类型约束，好处是：编译时，（IDE)能提前发现出错误。避免人工错误。对常用到的字符串，放到常量类中定义。另外一个好处，ide自动提示也比较友好。而对比字符串，则需要人工的复制或输入。

java的类型，比较多、复杂。基础类型，比如int，其他类型，类，接口、装饰器、枚举等等。在java里面，一切皆对象，但是呢，却又有int这种类型。又导致了包装类型的出现。（因为有些类型必须要出现）。而包装类型跟基础类型的转换，又出现了自动装箱、自动拆箱的概念（实际上就是编译器偷偷做的工作）

符号不能重载。结果==就是使用equals()函数。没有符号直观。

泛型，擦除法，不是在jvm层面的解决方式，功劳在编译器。它做得类型验证等等。只是省去了人去转换类型。

注解，本质是类似于注释一样的东西，只不过可以选择在那个阶段保留下来。注解，本身没有任何作用，使注解发挥作用的是代理proxy。在运行时增加。好处，加个注解，能实现强大的切入增加功能。缺点：实现逻辑相分离，实现原理不易查找。

spring功劳，IOC（依赖注入容器）、AOP切面编程。这确实非常强大，springboot中，所有的组件像积木一样，可以灵活的搭配、更换。像机器人一样，各个部件，不满易可以自定义更换。

springboot，类似于其他系统中的框架一样，将一些常用的各种组件封装起来，约定大于配置。通过导入依赖，在配置中，适当配置，更高级的是，定义@Configuration,自定义BEAN。自动配置，最大的功劳莫过于各类的starter。而starter就是利用一些条件配置实现的。（简单理解，你不配置、没有提供相关的类的实例时，ok，我来提供，然后你直接拿来用即可。）springboot确实优秀，能灵活搭配、更换各种组件。额外话，springboot如何保持各组件的兼容性呢？确实存在，升级了某些组件，由于冲突，项目无法启动的情况。大概为了解决这种冲突，官方有springboot parent来管理，不同版本的springboot使用的相关的组件，官方提供的组件，貌似是稳定、兼容。但是，其他第三方的组件，貌似不会太稳。

接口，作用也很大。不负责具体的实现，但是呢，却能约束调用与被调用双方，按通一规则来。面向接口编程。典型的例子,JDBC它只是一个接口，没有实现，具体的mysql等等实现改接口即可。但是，对调用方来讲，抹平了调用的差异，一套代码换任何实现，都没有问题。其他，还有JMS等等。

面向对象

java中的类型太多了，不易让人理解。比如，它有基本类型、有对象、有接口、有枚举类、有注解、还有记录类等等，他们都表是一种类型。因为java是严类型约束的系统。造成的困惑：基础类型跟包装类型的自动转换（是在编译器的帮助下自动转换的）。注解也是类，就是打标记。

 

1） 封装：指能够把一个实体的信息、功能、响应都装入一个单独的对象中的特性。 2） 继承：继承的方法允许在不改动原程序的基础上对其进行扩充，这样使得原功能得以保存，而新功能也得以扩展。这有利于减少重复编码，提高软件的开发效率。 3） 多态：同一操作作用于不同的对象，可以有不同的解释，产生不同的执行结果。在运行时，可以通过指向基类的指针，来调用实现派生类中的方法。 4）抽象：抽象(Abstraction)是简化复杂的现实问题的途径，它可以为具体问题找到最恰当的类定义，并且可以在最恰当的继承级别解释问题。

反射

反射主要是另外一种方式的代码调用。对于php这种语言，也有反射，但是它反射没有那么负责，其实就是提供了一些偏底层的函数，然后可以让你获取到一个类的方法、变量等等，然后调用。java呢，其实也是差不多的，但是呢，它是以类的封装进行的。

反射存在的意义是：另辟蹊径，以特殊的方式，在不确定的情况下，进行自醒，打破约束，进行非常规的顶用。比如定义一个函数，参数呢，比如是Object，不确定是合种类型，但是呢，依然能获取到对象的属性、调用到对象的（甚至是私有）方法。

java反射的基础：Class。每个类装载的时候，都会伴生一个Class对象，该对象大概描述了该类的一些属性。（相当于是记录者）

         * start a new thread with the given command as its first
         * task.  The call to addWorker atomically checks runState and
         * workerCount, and so prevents false alarms that would add
         * threads when it shouldn't, by returning false.
         *
         * 2. If a task can be successfully queued, then we still need
         * to double-check whether we should have added a thread
         * (because existing ones died since last checking) or that
         * the pool shut down since entry into this method. So we
         * recheck state and if necessary roll back the enqueuing if
         * stopped, or start a new thread if there are none.
         *
         * 3. If we cannot queue task, then we try to add a new
         * thread.  If it fails, we know we are shut down or saturated
         * and so reject the task.
         */
        int c = ctl.get();
        if (workerCountOf(c) < corePoolSize) {
            if (addWorker(command, true))
                return;
            c = ctl.get();
        }
        if (isRunning(c) && workQueue.offer(command)) {
            int recheck = ctl.get();
            if (! isRunning(recheck) && remove(command))
                reject(command);
            else if (workerCountOf(recheck) == 0)
                addWorker(null, false);
        }
        else if (!addWorker(command, false))
            reject(command);
    }

发版、定时任务

项目发版的时候，要保证服务的稳定性，有几种方式。1、在容器化部署的时候，通过控制，新容器起来时，才会删除旧的容器，服务是稳定的。2、非上云项目，可以脚本启动新服务，确定新服务稳定后，再删除旧服务。3、依赖中间件解耦。即，与其他服务对接的时候，并非直接通过接口，而是中间件（比如存redis、csv文件等等）。而中间件基本上是稳定、不会升级的。这样也能保证稳定性。

定时任务，并发的锁。1、使用mysql锁、redis锁等锁方式，来锁定。（比如shedlock，通过注解）2、定时任务通过http + curl触发，保证只有一个地方触发（但是确点是，如果布置了多个，则也会触发多个）,定时服务器可以单独部署。

方式2，貌似实现简单，不需要多锁控制，但是嘛，需要注意不要部署多个，而且后期排查过程中，由于分离部署，可能略麻烦。方式1，编码上略微复杂一些。

构造器注入

参考

构造器注入的好处：1、可以使用使用private final格式修饰成员变量，不可变性。2、相关的属性，在构造的时候，需要完全注入，才会完成初始化工作。3、

1、依赖不可变：其实说的就是final关键字，这里不再多解释了。不明白的园友可以回去看看Java语法。 2、依赖不为空（省去了我们对其检查）：当要实例化FooController的时候，由于自己实现了有参数的构造函数，所以不会调用默认构造函数，那么就需要Spring容器传入所需要的参数，所以就两种情况：1、有该类型的参数->传入，OK 。2：无该类型的参数->报错。所以保证不会为空，Spring总不至于传一个null进去吧 😦 3、完全初始化的状态：这个可以跟上面的依赖不为空结合起来，向构造器传参之前，要确保注入的内容不为空。

servlet-单例模式

servlet并不会为每个http请求创建实例，而是多个http请求对应同一个servlet。通俗的讲呢，servlet是单例模式，并不会随着http请求，返回的创建serlet，至于有多少个servlet，这个视情况而定。因为在ssm框架的时候，完全可以自定义请求对应url，然后servlet来响应请求。而使用springboot，默认的，其实创建了一个前端控制器DispathServlet，用来响应所有的请求，除此之外，还可以自定义servlet来响应。

Servlet是单例模式，是线程不安全的，故，不要访问对象的属性来保存值。即

xxxxxxxxxx
public class ExampleServlet extends HttpServlet {
    private Object thisIsNOTThreadSafe;//在方法外声明变量并在方法中使用很危险！！！
}

几个主要的方法：Service,doGet,doPost

  1. doGet()：请求数据会以？的形式拼接在请求头中，不安全，没有请求实体部分。另外，浏览器对URL的长度有限制，所以get请求不能携带大量数据。
  2. doPost()：数据不会显示在URL中，而是单独划分一块区域用于保存数据。POST 比 GET 更安全，因为参数不会被保存在浏览器历史或 web 服务器日志中。
  3. Service()：POST和GET底层仍为Service方法，所以一般情况下，重写Service方法即可。当需要调用时，指明需要哪种方法即可，无需专门编写POST和GET。

Qualifier

Qualifier

默认情况下，@Autowired 按类型装配 Spring Bean。 如果容器中有多个相同类型的 bean，则框架将抛出 NoUniqueBeanDefinitionException， 以提示有多个满足条件的 bean 进行自动装配。

通过使用 @Qualifier 注解，我们可以消除需要注入哪个 bean 的问题。@Qualifier 注解与我们想要使用的特定 Spring bean 的名称一起进行装配，Spring 框架就能从多个相同类型并满足装配要求的 bean 中找到我们想要的，避免让Spring脑裂。

备注：按类型注入，一般是按接口类型来区分。

ControllerAdvice

ControllerAdvice和RestControllerAdvice(继承前者)，主要是控制器增加，用来处理全局异常、增加全局数据等等。搭配@ExceptionHandler、@ModelAttribute以及@InitBinder使用。

二者区别，类似于@RestController与@Controller的区别，@RestControllerAdvice注解包含了@ControllerAdvice注解和@ResponseBody (配置的视图解析器InternalResourceViewResolver不起作用，返回的内容就是Return里的内容)注解。 

xxxxxxxxxx
@RestControler = @Controller + @ResponseBody
@RestControllerAdvice = @ControllerAdvice + @ResponseBody.
